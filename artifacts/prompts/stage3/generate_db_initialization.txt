## Critical Rules

1. **Status / state compatibility.** Each entity's status must satisfy the precondition of every action targeting it (e.g. cancellable-class status before a cancel; "Active" before suspend; "Overdue" before a payment request; the OPPOSITE state before a toggle). The INITIAL DB satisfies the precondition of the FIRST action; later transitions are produced by the actions themselves.
2. **Referential integrity.** Every cross-reference resolves: action-argument IDs exist in the right collection (or are runtime-created -- see rule 7); "owns" arrays (`user.reservations`, `user.orders`, `customer.line_ids`, `customer.bill_ids`) match the corresponding back-references; nested references (`reservation.user_id`, `bill.customer_id`, `line.plan_id`, `line.device_id`, item-variant references) resolve to real entities.
3. **No required field is null.** Use empty containers/strings/zeros only where the type allows them; never `null` for required non-nullable fields.
4. **Enums are exact.** Use literal schema values for every status/type/brand/mode/category. Inventing values fails validation.
5. **Numeric types match the schema.** Integer-typed fields use whole numbers (and so do prices when the policy implies integer arithmetic); floats only where the schema allows them.
6. **Economic consistency.** Bill `total_due` (or equivalent) equals the sum of its line items. Payment-history amounts reconcile with item prices plus all applicable policy fees (insurance, non-free baggage, change fees, etc.). Gift-card / certificate balances cover what they pay for; certificates are restricted per the policy. Recompute from first principles -- "what total would a policy-following agent charge?" must match.
7. **Dynamic entities are NOT in init.** When a tool mints IDs at runtime from a fixed pool (newly booked reservations, fresh certificates, etc., listed in the policy/spec), do NOT include those entities; the runtime creates them. If a later action references one by ID, that argument MUST use the exact runtime-assigned ID.
8. **Dual-DB consistency (when both DBs are present).** User-DB identifiers (e.g. device phone number) match an entity on the agent side. User-DB mirrored flags (line-active, roaming-allowed, data-exceeded) reflect the backend value. The user-DB device state must MATCH the symptom the user reports; if a corrective user action toggles a setting, the initial value must be the OPPOSITE of the toggle target (otherwise the action is a no-op).
9. **Policy-dependent setup.** When the policy requires a specific condition (membership tier, insurance flag, contract date, prior history, eligibility window) for an action to succeed, establish that condition on the targeted entity. If the policy lists alternative conditions, pick one and make it hold.
10. **Realistic free-parameter values.** When the action sequence has free parameters (default baggage count, insurance flag, payment method choice), pick what an agent would reasonably use given (a) what the user actually says in conversation and (b) policy defaults when the user does not state otherwise. If the task expects a non-default value, the user's instructions must direct the user to state it proactively.
